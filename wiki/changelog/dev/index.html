    Changelog (Developer / Technical)¶

    Full technical detail for each release. Companion to CHANGELOG.md — same section structure, same entry numbering, same bold lead per entry, but the body keeps class names, file paths, audit IDs, error fingerprints, and failed-approaches notes that get trimmed from the user-facing log.

    Lift from commit-message bodies on each release; this file is a curated rollup, not new authoring.

    Legend: 🟢 confirmed working in-app  ·  🔴 added but not yet verified in-app

    Fixed¶

The except (TypeError, ValueError) didn't catch it, so the whole apply_colormanagement_settings aborted. Fix: drop the per-curve 'extend' field at both ends. Parent-level extend already round-trips via the scalar view_settings.curve_mapping.extend path captured by the parent attribute loop. Legacy presets carrying the stale per-curve key are silently ignored on apply. See doc/troubleshooting/colormanagement_preset_curvemap_extend.md.
    2. tks.tag_add_new no longer hard-crashes Blender via unbounded Python recursion. The _preset_auto_pending guard correctly suppressed re-scheduling within the same 0.2 s window, but as soon as the deferred timer fired and cleared the guard, the next draw() (which runs on every redraw — mouse move, frame change, …) saw dirty_types still non-empty and re-scheduled. Net: revert_all_presets ran roughly every 0.2 s as long as the panel was visible. Fix: move DROP_REAPPLY off the draw trigger and onto the event bus. Added _on_vl_switch_post_drop_reapply subscriber in panels/navigation/logic.py and register_drop_reapply_handler / unregister_drop_reapply_handler wired through panels/navigation/__init__.py::register(). Same bug pattern as Fixed #2 (tks.tag_add_new): commit f0da905 did a careless text-rename and pointed the single-tag delete dispatcher in TKS_OT_DeleteTagItem.execute (operators/tags/ops_groups_query.py:262) at the operator's own bl_idname (return bpy.ops.tks.delete_tag_item()), replacing the original target bpy.ops.tks.roadmap_remove_tag when the roadmap subsystem was removed.     Added¶

    1. Camera Visibility Automation — only the active VL's camera is visible, hands-free. A new automation under Preferences → Workflow → Automations → Camera Visibility Automation keeps the outliner and viewport clean across view-layer switches. The master toggle activates the behavior; three granular sub-flags configure it: Show Assigned Camera clears hide_viewport, hide_render, and the per-VL eye icon on the cascade-resolved camera every time the view layer changes — even if the user had manually hidden it. Hide Other Cameras stamps the same three flags TRUE on every other CAMERA object in the active scene, so non-assigned cameras disappear from outliner, viewport, and render. When the camera is linked into multiple collections, the first host whose ancestor chain is already fully visible wins; if none qualifies, the first host found is used. The whole feature is hooked into the cascade hot-path right after scene.camera = camera, short-circuits with zero cost when the master toggle is off, and persists across Blender restarts via addon prefs autosave. The icon-state cache _vl_cascade_cache in panels/context/draw_layers.py was only invalidated by the Scene / VL / Global update= callbacks (via _invalidate_and_redraw()) plus the ~200 ms context sync timer. Assigning at SG / VLG / VLV tiers — which don't have update callbacks — left the stale entry in place; the icon kept its old (often red/broken) color until the next sync tick. TKS_OT_SetCascadeProp.execute() now calls invalidate_vl_cascade_cache() plus signal_redraw() after every picker assignment regardless of prop type, since all four pickers (ACTION / WORLD / CAMERA / OUTPUT_RULE) share the same icon-state cache.
    8. Active takes-tree row no longer jumps to the last item after a cascade picker assignment. Assigning a value through the generic picker setter could leave wm.tks_all_viewlayers_index pointing somewhere unexpected — most visibly with ACTION assignments where the property write triggers depsgraph.update() + a context-sync-driven structural rebuild whose selected-key lookup occasionally mis-resolves. The picker click is meant to set a value on the row the user clicked, never to move the selection. TKS_OT_SetCascadeProp.execute() now snapshots the active tree index before setattr, runs the cascade refire + cache invalidation + redraw signal, and restores the saved index if it drifted (bounds-checked against the possibly-rebuilt collection, so we don't fight the system when the row genuinely vanished). Defensive — fixes the symptom for all four picker prop types (ACTION / WORLD / CAMERA / OUTPUT_RULE) regardless of which sync path is shuffling the index underneath.
    9. Scene / VL / VLG / VLV camera picker only shows cameras linked to the target scene, so the icon never stays red after a "successful" assignment. Previously the camera picker listed every CAMERA object in the file via _PICKER_CONFIG['CAMERA'].collection_fn. If the user picked a camera that lived only in another scene's collection hierarchy, the cascade engine's per-scene name in scene.objects validation in apply_overrides() (core/cascade.py:500-523) silently auto-cleared the assignment — the user saw the value briefly in the picker, then watched the cascade icon stay red because there genuinely was no usable camera at any resolved tier afterwards. New _camera_collection_fn(context) reads the picker's stashed tks_picker_scene_name + tks_picker_target_type and pre-filters to cameras in scene.objects when the target is scene-bound (Scene / VL / VLG / VLV). Scene-agnostic tiers (GLOBAL, SG) still see the full list because their cascade fans out to many scenes; the per-scene validation correctly happens at apply time there. The cascade's silent auto-clear safety net remains for stale assignments loaded from older .blend files or external scripts.

    [0.7.2] - 2026-05-13¶

    Added¶

    1. Keyframe pie menus on I and Alt+I — both off by default. Two opt-in pies. The Keyframe Pie (I) covers insert/set actions (Set as Rest Default, Insert at Rest State, Snap to Rest, Snap All to Rest, Insert Keyframe (Native)), and the Clear Pie (Alt+I) covers clear/remove actions (Clear Current Keyframe, Unset Rest Key (Channel), Unset Entire Rest Slot, Clear All Keyframes (Object), Clear All Rest Keys (VL), Delete Keyframe (Native)). Each pie has its own 8-slot configurator under Preferences → Workflow → Pie Menu with mutual-exclusion and reset workflow matching the Navigation and F12 pies. Enable each master toggle independently; until you do, I and Alt+I behave exactly like stock Blender.
    2. Pie hotkeys are now rebindable from the configurator. The center cell of every pie configurator (Navigation, F12 Render, Keyframe, Clear) is a live hotkey button — click it and Blender's "press a key" prompt captures a new key + modifier combo. Want to keep native I behavior and put the Keyframe Pie on a different key? Click the center cell and press your preferred chord. The same widget works for the Ctrl+Shift+C navigation pie and the F12 render pie. A reset icon (↩) sits glued to the hotkey field as a connected sibling button — one click restores the built-in default (Ctrl+Shift+C, F12, I, Alt+I) with a "Reset hotkey to default" tooltip. The whole rebind widget is centered between the W and E slot dropdowns and falls back to a static label when the pie's master toggle is off (no keymap entry to edit yet).
    3. Hotkeys panel lists all four pie shortcuts. The Navigation, F12 Render, Keyframe, and Clear pie entries now appear inline in Preferences → Workflow → Hotkeys alongside the other Takes shortcuts, so you can review or rebind every pie from one place with the standard Blender keymap-editor controls (expand row, change modifiers, restore default).
    4. Editor Scope for the keyframe pies covers every keyframe-capable editor. Per-editor checkboxes under Preferences → Workflow → Pie Menu → Editor Scope let you opt the pies into the 3D Viewport, Properties, Graph Editor, Dope Sheet, NLA Editor, Shader Editor, Image Editor, Movie Clip Editor, Video Sequencer, and Outliner. The wrapper routes per context.area.type and falls back to the editor's native I / Alt+I (anim.keyframe_insert, anim.keyframe_insert_button, graph.keyframe_insert, clip.keyframe_insert, …) when an editor's toggle is off, so disabled editors keep native behavior.
    5. Hover-aware slot visibility. Outside the 3D Viewport, hover-dependent slots (anything that needs a specific property — Set/Insert at Rest, Snap to Rest, Clear Current Keyframe, Unset Rest Key (Channel), Insert/Delete Keyframe (Native)) only appear when the cursor is over a keyframable value. Empty canvas hides them and leaves the slot direction as a separator, so muscle memory still works for the 8 directions. The 3D Viewport always shows every slot — selection-based actions don't need hover.
    6. Three new Rest State operators. Unset Rest Key (Channel) removes a single fcurve's channel for the active property, Unset Entire Rest Slot removes the active object's slot from the Rest State action, and Clear All Rest Keys (VL) walks every object in the active view layer and clears their rest slots. All three appear as slot choices in the Clear Pie.
    7. Viewport tool settings now sync across scenes. A new Workflow → Sync → Viewport Tool Settings section lets you mirror selected 3D viewport tool settings — transform orientation + pivot + Affect-Only toggles, snap settings, proportional editing, mirror, auto-merge, lock-object-mode, UV — across every scene as you change them. Last-write-wins: any scene's change to a synced setting propagates to all the others. Each group has its own toggle, gated by a master Enable Viewport Sync switch (off by default). The msgbus subscriptions are only armed while the master is on.
    8. Auto-clearing the Scene World on an empty cascade is now optional. A new Auto-clear World when Unassigned toggle under Preferences → Workflow → Automations (on by default) controls whether scene.world gets wiped when no world is assigned anywhere in the Takes tree. With it off, the existing Scene World is preserved and the Cascade Drift warning surfaces persistently as None → <WorldName> to remind you to put a world into the tree (or accept the manual assignment).
    9. Apply Scene World to any cascade tier from the drift warning. The drift row for World now opens a tier-picker dropdown (Global / Scene Group / Scene / VL Group / View Layer / VL Version) — pick a tier and the active scene.world is written into that level's tks_world override. Tiers that don't apply for the active context (no Scene/VL Group beyond the default "Ungrouped", no active VL Version) are greyed out. Replaces the single-target Accept button for World drift; Accept still appears for Compositor drift.
    10. Undo can no longer silently destroy a manually-assigned Scene World. A new Skip World-Clear in Post-Undo Cascade required-core strategy (on by default, under Preferences → Workflow → Debug → Undo-Redo) makes the post-undo apply_overrides skip the world-clear branch entirely. Before this, undoing back to a state where no world was in the cascade tree would wipe scene.world on the very next post-undo cascade run — even if the user had assigned it outside the tree.

    Changed¶

    1. The Inspector Watchlist keeps a stable height when it's empty. The list and its sidebar buttons (Add / Remove / Settings / Find Slot) now always render; the empty-state info message appears as a row in the stats slot above the list instead of replacing the whole list with an info box. The panel no longer jumps in height the moment the last watched object is removed.

    Fixed¶

    1. Pie hotkey rebinds now persist across every routine pref change. Earlier 0.7.2 builds rebuilt the pie keymap entries on every Editor Scope checkbox flip, master toggle round-trip, and addon reload — wiping any custom key the user had bound via the configurator's center hotkey button or via Blender's keymap editor. _register_keymap() is now idempotent across all three pies (Navigation, F12 Render, I + Alt+I) — it preserves existing kmi entries when found and only flips the active flag to match master state. Per-editor toggles and slot dropdown changes no longer touch the keymap layer at all (the wrapper operator reads prefs dynamically). Full keymap teardown only happens on addon unregister.
    2. New codebase audit catches scope-crossing writes early. Part J of the codebase stats (doc/reports/codebase_stats.md) now scans every Python file for writes to wm.keyconfigs.user and wm.keyconfigs.default and lists them grouped by severity. Lines tagged with a # SCOPE_OK: <reason> comment are exempt and still listed for review; anything unjustified deducts from the Health Score. The 17 legitimate user-keyconfig writes in 0.7.2 (Blender 5.x dispatch workaround, master-toggle reversible flips, full-teardown removes on unregister) all carry the marker now, so the audit reads 0 unjustified scope crossings.
    3. Pie menu keymaps no longer crash Blender on per-editor toggle. Flipping a checkbox in Editor Scope (or any path that re-runs pie keymap registration) could trigger an EXCEPTION_ACCESS_VIOLATION deep inside rna_KeyMap_item_remove → BLI_findindex when a stored kmi C-pointer had been freed by Blender between cycles. The unregister path now walks each keyconfig fresh and matches entries by idname + properties, so it never dereferences a stale pointer. The same hardening is applied to the Navigation, F12 Render Scope, Global Shortcuts, Tag Groups, and Context panel keymap teardowns.
    4. Ctrl+Shift+C navigation pie and F12 render pie fire reliably in Blender 5.x. Both pies registered only in the addon keyconfig and relied on Blender to mirror entries into the user keyconfig — which Blender 5.x doesn't do reliably, so dispatch silently failed even though DIAG showed entries active=True. Each pie now registers in both keyconfigs explicitly, the same fix used for the new I-key pies.
    5. The Post-Undo Cascade Ping no longer silently aborts on scenes without a compositor node tree. A null scene.node_tree access raised a non-fatal exception that bailed the entire ping mid-scan, so any datablock not yet visited never had its cascade rebuilt. The access is now guarded so the ping continues with the rest of the scan.
    6. The render-window-close watcher fires reliably during undo bursts again. The mutation-skip guard added in 0.7.1 broke the post-undo cascade ping — it relied on this timer firing promptly during undo. The guard is removed; a K5_EXEMPT code comment plus a project audit note keep it from being reintroduced.
    7. The F12 Render Scope pie shows operator errors as status-bar messages instead of console tracebacks. When the inner batch render operator raises a RuntimeError via self.report({'ERROR'}, …) — e.g. "Save file before starting Batch Render!" — the pie's dispatcher now catches it and re-reports the message via the pie operator's own status bar.
    8. Clean Empty Slots & Actions has a combined operator, a preview dialog, and a safety pass. A single Clean Empty Slots & Actions runs both passes after showing a preview of what will be removed; an extra safety pass leaves fake-user-pinned actions and slots that are still referenced elsewhere alone.
    9. JIT rest action cleanup runs in the two merge branches that previously skipped it. Two specific just-in-time merge paths left their source actions behind without scheduling them for removal — they now follow the same cleanup schedule as every other branch, so orphan actions stop accumulating after long sessions.
    10. Preset capture covers Hydra Storm engine settings and the Output Time Stretching ratio. Render presets now save scene.hydra.* (visible under Render Properties → Hydra when the render engine is Hydra Storm — currently just Export Method). Output presets now save the Time Stretching Old / New frame-mapping ratio. Both were silently absent before.
    11. Accepting a Cascade Drift no longer crashes on World or Compositor. accept_drift was assigning a World / NodeTree datablock to vl.tks_world / vl.tks_compositor, both of which are StringProperty. The latent type-mismatch was masked by the old auto-clear-world behavior never producing a drift in the relevant case; with the world-clear toggle the drift becomes reachable. Accept now writes actual.name and clears with "" to match the property type.
    12. The Context tree view now repopulates on every file load, not only the first of a session. The _sync_timer fast-path gate is driven by module-level globals in panels/context/handlers.py (_sync_dirty/_context_sync_dirty/_last_fingerprint/_last_structure_key). They start True (first load populates) then flip False after the first sync. Being Python globals they survive File>Open, but the WM-backed tks_all_viewlayers collection is reset by the load (Load UI swaps in the file's WM) → empty collection + closed gate + no re-arm = blank tree. The structural fallback only re-arms on a scene/VL count change (a same-structure reload doesn't trip it) and World-stored group data doesn't reliably fire depsgraph_update_post on load — the exact gap _handle_undo_redo_sync already covers for undo/redo. Fix: new @persistent _handle_load_post_sync (file-load twin of the undo handler) re-arms both dirty flags, nulls _last_fingerprint/_last_structure_key, and busts TKS_UL_AllViewLayers._sg_cache_token / groups._vs_cache_token; registered/unregistered on bpy.app.handlers.load_post in panels/context/__init__.py. Tests: dev/tests/test_tree_load_refire.py (5 — flag re-arm, fingerprint/structure-key clear, load_post arg tolerance, registration). 543/543 headless. See doc/troubleshooting/tree_blank_after_second_file_load.md.
